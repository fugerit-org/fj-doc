=== Eager vs Lazy initialization

By default, Fugerit Venus Doc has a lazy initialization approach, where document handlers are initialized only when first needed.

However, in some situations it could be beneficial to have an eager (Ahead Of Time) initialization, especially when:

* You want to catch configuration errors during application startup rather than at runtime
* You need to minimize the latency of the first document rendering operation
* You are running on frameworks with AOT (Ahead-of-Time) compilation like Quarkus native images, where lazy initialization can cause issues

This can be achieved using the built-in utility:

_org.fugerit.java.doc.base.config.InitHandler_

==== Async Initialization (Recommended)

For most scenarios, async initialization is recommended as it does not block the startup process:

[source,java]
----
package org.fugerit.java.doc.demo;

import io.quarkus.runtime.StartupEvent;
import jakarta.enterprise.context.ApplicationScoped;
import jakarta.enterprise.event.Observes;
import jakarta.inject.Inject;
import org.fugerit.java.doc.base.config.InitHandler;

@ApplicationScoped
public class AppInit {

    @Inject
    DocHelper docHelper;

    void onStart(@Observes StartupEvent ev) {
        /*
         * This will initialize all the doc handlers using async mode.
         * The initialization happens in the background without blocking startup.
         */
        InitHandler.initDocAllAsync(
                docHelper.getDocProcessConfig().getFacade().handlers() );
    }

}
----

==== Sync Initialization

In scenarios where you need to ensure all handlers are fully initialized before the application becomes ready to serve requests, use synchronous initialization:

[source,java]
----
void onStart(@Observes StartupEvent ev) {
    /*
     * This will initialize all the doc handlers synchronously.
     * Startup will be blocked until all handlers are ready.
     * Use this when you need guaranteed initialization before accepting requests.
     */
    InitHandler.initDocAll(
            docHelper.getDocProcessConfig().getFacade().handlers() );
}
----

**When to use sync initialization:**

* Health check endpoints need to ensure handlers are fully initialized
* You want to fail fast if there are configuration issues
* Your application requires predictable startup timing
* You are running in an AOT environment where lazy initialization is problematic

==== Single Handler Initialization

You can also initialize a specific handler instead of all handlers:

[source,java]
----
import org.fugerit.java.doc.base.config.InitHandler;void onStart(@Observes StartupEvent ev) {
    DocHandler docHandler = findHandler();

    // Initialize only the PDF handler
    InitHandler.initDoc(
        docHandler
    );

    // Or synchronously:
        docHandler
    );
}
----

NOTE: This will initialize the specified doc handlers by attempting to create a link:https://github.com/fugerit-org/fj-doc/blob/main/fj-doc-base/src/main/resources/config/init_doc/doc-init.xml[simple test document]. This approach validates that the handler configuration is correct and all required dependencies are available.
